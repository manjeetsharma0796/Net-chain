The problem

B2B netting today is a
privacy nightmare.

Every counterparty in a multilateral net sees the full exposure table — who owes whom, how much, and when. That's your trade data, your credit risk, your negotiating position. All of it, visible to your competitors.

Today

Shared spreadsheet

Every participant sees every obligation. Full counterparty exposure. Centralised trust.

NetChain

Confidential settlement

A cannot read B's obligations. The ledger enforces this — not the UI. One operator, known and authorised. No MPC needed.

Win 1 · Counterparty Privacy

C cannot see A↔B.
The ledger says so.

The A→B Obligation has signatory obligor, observer obligee + operator. C is not listed — so the node won't even acknowledge the contract exists.

Win 4 · On-Ledger Policy Gate

The policy lives
on the ledger, not the UI.

Each party holds a TreasuryPolicy with a maxSettlementPerCycle cap. The CheckSettlement choice asserts it — an over-cap attempt is rejected inside the Canton transaction. An agent cannot route around it.

# POST /api/ledger/policy-check { party: "company-c", amount: 250000 }
← 200 { ok: false, ruleFired: "TreasuryPolicy breach: 250000.0 > cap 200000.0" }

# POST /api/ledger/policy-check { party: "company-c", amount: 150000 }
← 200 { ok: true }

Privacy model — the honest version

Counterparties are blind.
Operator is known + authorised.

What's private

A↔B obligations are invisible to C — enforced at the ledger node, not the UI. Each party reads only its own NetPosition. No counterparty can access another's net figure.

What the operator sees

The netting operator (signatory on NettingCycle) sees all obligations in-scope for the cycle. This is the settlement bank model — a known, regulated coordinator, not a hidden intermediary.

Future: MPC / ZK

A blind operator is possible with MPC or ZK proofs of netting correctness — explicitly deferred. The architecture makes this a future layer, not a rework.

Why Canton

Privacy is a first-class
ledger primitive.

Sub-transaction privacy

Parties receive only the sub-transactions they're a stakeholder in. Privacy is enforced at the node — not the application layer.

Atomic by construction

Multi-leg settlement in one Daml choice = one Canton transaction. Either every leg clears, or the whole thing aborts. No partial state.

Known-entity identity

Parties are real legal entities, not pseudonymous keys. TreasuryPolicy + allowed-counterparties = enforceable contractual rules.
